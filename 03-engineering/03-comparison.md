# 阶段 3.3 · 方案对决：本体 vs 传统 RAG

> **一句话**：本体治"准确率"，RAG 治"灵活性"
> **目标**：5 分钟能选对方案

## 1. 一句话总结

| 方案 | 准确率 | 灵活性 | 成本 | 适合 |
|------|--------|--------|------|------|
| **RAG（向量）** | 60-90% | 高 | 低 | 文档问答、灵活场景 |
| **本体 + SPARQL** | 95-100% | 低 | 中 | 决策、客服、跨源 |
| **GraphRAG** | 80-92% | 中 | 高 | 大型 RAG、复杂场景 |

## 2. 生活类比

```
RAG     = 翻书找答案
本体    = 查数据库
GraphRAG = 先翻书，再查数据库交叉验证
```

**翻书**：快、可能找错。
**查数据库**：慢、一定对。
**交叉验证**：慢但准。

## 3. 一个具体案例

**问题**："找出 2025 年买过 iPhone 且最近 30 天没买任何手机的客户"

### 3.1 RAG 的灾难

```
→ LLM 看到 100 个 "iPhone" 文本片段
→ 完全没有"购买"结构
→ 答案：编造客户名、编造日期
→ 准确率：0%
```

**为什么不行**：RAG 看的是"文本相似度"，看不到"关系"。

### 3.2 本体的优雅

```sparql
PREFIX ex: <http://shop.com/>
SELECT DISTINCT ?cust WHERE {
    ?cust ex:purchased ex:iPhone15 .
    FILTER NOT EXISTS {
        ?cust ex:purchased ?other .
        ?other a ex:Phone ;
               ex:purchaseDate ?d .
        FILTER (?d > "2025-05-09"^^xsd:date)
    }
}
```

**准确率：100%**。因为"购买"是显式的三元组。

### 3.3 GraphRAG 的折中

```
→ 先用向量检索召回 50 个候选客户
→ 再在知识图谱里做精确过滤
→ 准确率：80-90%，延迟 1.2s
```

## 4. 5 维度对比

| 维度 | 传统 RAG | 本体 + SPARQL | 混合（GraphRAG） |
|------|----------|---------------|------------------|
| **结构化查询准确率** | 0-30% | 95-100% | 70-90% |
| **自然语言准确率** | 70-90% | 60-80% | 75-92% |
| **可解释** | ❌ 黑盒 | ✅ SPARQL 可回放 | ⚠ 部分 |
| **延迟** | 200-500ms | 50-300ms | 500-1500ms |
| **每千次成本** | $1-3 | $0.5-2 | $2-5 |
| **数据更新** | 容易 | 难 | 中 |
| **数据规模** | 文档 | 结构化数据 | 大表 |
| **学习曲线** | 低 | 高 | 中 |
| **可推理** | ❌ | ✅ | ⚠ |

## 5. 5 个场景的选择

| 场景 | 推荐 | 原因 |
|------|------|------|
| 通用聊天 | RAG | 没结构化数据 |
| 客服（订单查询） | **本体** | 订单是结构化的、要准 |
| 文档问答 | RAG | 文档本身没结构 |
| 决策支持 | **本体** | 不允许错 |
| 跨组织数据 | **本体** | 双方要共同 schema |
| 多 Agent 协作 | **本体** | Agent 共享世界模型 |
| 大型 RAG | GraphRAG | 准确率 vs 成本平衡 |

## 6. 决策树

```
              你的数据有结构化 schema 吗？
              (有数据库 / API / 知识图谱)
                          │
            ┌─────────────┴──────────────┐
            │ 是                          │ 否/纯文本
            ▼                             ▼
    准确率要求 95%+ 吗？              走 RAG（快速上手）
            │
    ┌───────┴────────┐
    │ 是              │ 否
    ▼                 ▼
  本体 + SPARQL     RAG（成本低、灵活）
```

## 7. 成本对比（10 万次查询/月）

| 方案 | 一次性投入 | 月运营 | 12 月总成本 |
|------|-----------|--------|-------------|
| RAG | 0.5 人月 | $300 | $4,000 |
| 本体 + SPARQL | 6 人月 | $100 | $36,500 |
| GraphRAG | 3 人月 | $600 | $19,800 |

**本体前期贵**——但**准确率 95% vs 30%** 带来的业务价值远超成本。

## 10. 避坑提示

- ❌ **别"非此即彼"**：实际项目常 RAG + 本体混用
- ❌ **别只算"准确率"**：成本、延迟、维护成本都算
- ❌ **别在错误场景用本体**：纯文本上本体是过度设计
- ✅ **RAG 起步，必要时升级到本体**
- ✅ **结构化数据用本体，文档用 RAG**
- ✅ **混合方案最实用**：意图分类分发
